2. ML Models
Regression Models
HOT · L1 vs L2 cold
 INTERMEDIATE
Linear RegRidge (L2)Lasso (L1)RF Regressor
MAEMSERMSER²Adj R²MAPE
Ridge (L2): shrinks all weights toward zero, none exactly zero — use when many small features contribute. Lasso (L1): drives some weights to exactly zero — built-in feature selection. MAE = outlier-robust; RMSE = punishes large errors; R² = variance explained. Use Adj R² when comparing models with different feature counts.
2. ML Models
Cross-Validation & Bias-Variance
HOT · K-fold every ML round
 DEEP
K-Fold CVStratified K-FoldSMOTE
 — N/A — High variance (overfit): train F1 high, val F1 low → regularise, more data, reduce max_depth. High bias (underfit): both errors high → more features, complex model. Stratified K-Fold always for imbalanced datasets — preserves class ratio. Imbalance fix order: class_weight='balanced' → SMOTE (train only) → lower threshold for minority class.
2. ML Models
Feature Scaling & Normalization
HOT · Asked in every ML round
 DEEP
StandardScalerMinMaxScalerRobustScalerNormalizer
 — N/A — StandardScaler: mean=0, std=1 — use for algorithms sensitive to scale (LogReg, SVM, Neural Nets). MinMaxScaler: scales to [0,1] — use when bounded range needed. RobustScaler: uses median & IQR — robust to outliers. When NOT to scale: tree-based models (RF, XGBoost) — they're scale-invariant. Critical: fit scaler on train only, then transform train & test — never fit on test (data leakage).
2. ML Models
Feature Engineering
HOT · Differentiates candidates
 DEEP
One-Hot EncodingLabel EncodingTarget EncodingFeature Interaction
 — N/A — One-Hot Encoding: categorical → binary columns — use for nominal categories (color, city). Label Encoding: ordinal categories only (low/med/high). Target Encoding: replace category with mean of target — beware overfitting, use with CV. Feature interactions: multiply/combine features (age × income). Datetime features: extract hour, day_of_week, month, is_weekend. Handle missing: mean/median imputation or create is_missing flag.
2. ML Models
Hyperparameter Tuning
HOT · Production ML essential
 INTERMEDIATE
GridSearchCVRandomizedSearchCVOptunaBayesian Optimization
 — N/A — GridSearchCV: exhaustive search over param grid — slow but thorough. RandomizedSearchCV: random sampling — faster, good for large spaces. Key params: XGBoost (max_depth, learning_rate, n_estimators), RF (max_depth, min_samples_split, n_estimators). Always use CV inside tuning to avoid overfitting. Optuna: modern Bayesian approach — learns from previous trials, faster convergence.

Forward pass: input × weight + bias → activation → output. Activations: ReLU (hidden), Sigmoid (binary out), Softmax (multi-class). Backprop = chain rule gradient. 3. DL / NLP
BERT vs GPT vs T5
HOT · 30-sec answer expected
 INTERMEDIATE
BERTGPTT5/BARTSentence-BERTDistilBERT
 — N/A — BERT (encoder, bidirectional): sees full context → classification, NER, sentiment. GPT (decoder, autoregressive): generation, summarisation, chatbots. T5/BART (enc-dec): understands AND generates → translation, summarisation. Sentence-BERT: fixed-size vectors → semantic search. Why BERT not GPT for sentiment? Bidirectional context = better understanding.

4. GenAI & RAG
Fine-Tuning: LoRA & QLoRA
HOT · PEFT in every senior JD
 DEEP
LoRAQLoRAAdapterUnslothHuggingFace TRL
 — N/A — LoRA: freeze 7B base model, add low-rank A×B matrices (rank=8) → only ~4M params trained vs 7B full. QLoRA: adds 4-bit NF4 quantisation → 7B model fits in 4GB VRAM. Rule: fine-tune for style/format consistency; use RAG for knowledge — never fine-tune to add new facts. When asked "how would you fine-tune without LangChain?" — explain the primitives.

6. LLMOps
Model Monitoring in Production NEW
HOT · New baseline in 2026 JDs
 INTERMEDIATE
MLflowLangfuseLangSmithEvidentlyAIRedis
Latency/reqToken cost/queryRAGAS FaithfulnessUser feedback
Why LLMOps: LLMs degrade silently — queries shift, responses worsen, no obvious error log. Track: latency, token cost, thumbs-up/down, answer relevance. Drift detection: if avg RAGAS faithfulness drops week-over-week → trigger RAG pipeline review. Cost control: set max_tokens, cache frequent queries in Redis (TTL=24hrs), batch embed. Fallback: if OpenAI fails → serve cached answer or route to local Llama — never return 500.
6. LLMOps
Cost Optimisation & Rate Limits
HOT · Production JDs ask this
 INTERMEDIATE
RedisCeleryQuantizationBatching
Cost/1k requestsP95 Latency
Cache frequent queries in Redis. Batch embed documents, not one-by-one. Quantization: 4-bit inference reduces VRAM, speeds up throughput. Rate limit handling: exponential backoff, queue with Celery. A/B test prompts before rolling out. When asked "LLM costs are too high": reduce max_tokens, cache, switch to smaller model for simple queries, use MoE (Mixture of Experts) routing.
